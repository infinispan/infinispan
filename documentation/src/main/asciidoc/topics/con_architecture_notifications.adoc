= Notifications and listeners

{brandname} provides a publish-subscribe notification mechanism that allows applications to receive callbacks when cache entries are created, modified, removed, expired, or evicted.
Listeners can be local (receiving events only from the node they are registered on) or clustered (receiving events from all nodes in the cluster).

== Event model

Cache events are fired at two points during an operation: _before_ the operation executes (PRE) and _after_ it completes (POST).
All events implement the `Event` interface and carry a reference to the cache, the event type, and the PRE/POST flag.

== Listener registration

A listener is any object annotated with `@Listener` that has one or more methods annotated with event type annotations.

[source,java]
----
@Listener
public class MyListener {

   @CacheEntryCreated
   public void onCreated(CacheEntryCreatedEvent<String, String> event) {
      if (!event.isPre()) {
         // entry has been created
      }
   }

   @CacheEntryModified
   public void onModified(CacheEntryModifiedEvent<String, String> event) {
      // ...
   }
}

cache.addListenerAsync(new MyListener());
----

The `@Listener` annotation supports several attributes:

[cols="2,1,4"]
|===
|Attribute |Default |Description

|`sync`
|`true`
|When `true`, the listener is invoked synchronously in the calling thread and the cache operation blocks until the listener returns. When `false`, the listener is invoked asynchronously in a separate notification thread pool.

|`clustered`
|`false`
|When `true`, the listener receives events from all nodes in the cluster, not just the local node.

|`includeCurrentState`
|`false`
|When `true`, all existing cache entries are delivered as `CacheEntryCreated` events when the listener is first registered.

|`primaryOnly`
|`false`
|When `true`, only the primary owner of a key fires the event. When `false`, all nodes that process the operation fire the event.

|`observation`
|`BOTH`
|Controls whether the listener receives PRE events, POST events, or both.
|===

== Notification flow

The `CacheNotifier` dispatches events to registered listeners.
Events are generated by the interceptor chain during cache operations.

[plantuml, target=architecture-notification-flow, format=svg]
....
@startuml
skinparam defaultFontName sans-serif
skinparam shadowing false
skinparam sequenceMessageAlign center

participant "CacheImpl" as Cache
participant "Interceptor\nChain" as IC
participant "CacheNotifier" as CN
participant "Sync\nListener" as SL
participant "Async\nListener" as AL
participant "Notification\nThread Pool" as NTP

Cache -> IC: put(key, value)
activate IC

IC -> CN: notifyCacheEntryCreated\n(key, value, pre=true)
activate CN
CN -> SL: onCreated(PRE event)
activate SL
SL --> CN
deactivate SL
CN --> IC
deactivate CN

IC -> IC: Execute operation\non data container

IC -> CN: notifyCacheEntryCreated\n(key, value, pre=false)
activate CN
CN -> SL: onCreated(POST event)
activate SL
SL --> CN
deactivate SL
CN -> NTP: submit(POST event)
activate NTP
CN --> IC
deactivate CN
NTP -> AL: onCreated(POST event)
deactivate NTP

IC --> Cache: result
deactivate IC
@enduml
....

Synchronous listeners block the cache operation until they return, which is useful when the listener must execute as part of the same operation (e.g., updating a secondary index).
Asynchronous listeners are dispatched to a notification thread pool, so the cache operation completes without waiting.

== Filters and converters

Listeners can be registered with a `CacheEventFilter` to receive only matching events, and a `CacheEventConverter` to transform event data before delivery.

[plantuml, target=architecture-filter-converter, format=svg]
....
@startuml
skinparam defaultFontName sans-serif
skinparam shadowing false
skinparam activityBackgroundColor #f8f8f8

start
:Cache Event;
:CacheEventFilter.accept();
if (accepted?) then (yes)
  :CacheEventConverter.convert();
  :Deliver to Listener;
else (no)
  :Dropped;
  stop
endif
stop
@enduml
....

`CacheEventFilter`::
Evaluates each event against a predicate.
Only events where `accept()` returns `true` are delivered.

`CacheEventConverter`::
Transforms event data (key, old value, new value, metadata) into a custom payload.
Useful for reducing the data sent to remote listeners.

`CacheEventFilterConverter`::
A combined filter and converter for efficiency.
The `filterAndConvert()` method returns `null` to filter out events, or a converted value to accept and transform in a single pass.

Filters and converters are especially important for clustered listeners, where they reduce network traffic by evaluating on the event-producing node.

== Clustered listeners

A clustered listener receives events from all nodes in the cluster, not just the local node.
This is enabled by setting `@Listener(clustered = true)`.

[plantuml, target=architecture-clustered-listener, format=svg]
....
@startuml
skinparam defaultFontName sans-serif
skinparam shadowing false
skinparam sequenceMessageAlign center

participant "Application" as App
participant "Node A\n(registration)" as NA
participant "Node B" as NB
participant "Node C" as NC

App -> NA: cache.addListenerAsync(\nlistener, filter, converter)
activate NA

NA -> NA: Register listener locally
NA -> NB: ClusterListenerReplicateCallable\n(filter, converter, listenerId)
NA -> NC: ClusterListenerReplicateCallable\n(filter, converter, listenerId)

activate NB
NB -> NB: Install RemoteClusterListener\n(primaryOnly=true, POST only)
NB --> NA: OK
deactivate NB

activate NC
NC -> NC: Install RemoteClusterListener\n(primaryOnly=true, POST only)
NC --> NA: OK
deactivate NC
deactivate NA

...later: cache entry modified on Node B...

NB -> NB: RemoteClusterListener\nreceives local event
NB -> NB: Apply filter and converter
NB -> NA: ClusterEvent\n(filtered, converted)

NA -> NA: CacheNotifier delivers\nevent to original listener
NA -> App: listener callback
@enduml
....

The clustered listener flow:

. The application registers a listener on one node (the _origin_).
. The origin sends a `ClusterListenerReplicateCallable` to all other nodes, which installs a `RemoteClusterListener` on each.
. The remote listener is configured with `primaryOnly = true` so each event is captured exactly once (on the primary owner of the affected key).
. When a cache entry changes on a remote node, the `RemoteClusterListener` applies the filter and converter locally, then sends the resulting `ClusterEvent` to the origin node.
. The origin's `CacheNotifier` delivers the event to the original application listener.

Clustered listeners support four event types: `CacheEntryCreated`, `CacheEntryModified`, `CacheEntryRemoved`, and `CacheEntryExpired`.
Only POST events are delivered (not PRE).

If the origin node leaves the cluster, all `RemoteClusterListener` instances installed for that listener are automatically removed.

== Initial state transfer

When a listener is registered with `@Listener(includeCurrentState = true)`, {brandname} delivers all existing cache entries as synthetic `CacheEntryCreated` events before the listener starts receiving live events.

[plantuml, target=architecture-listener-initial-state, format=svg]
....
@startuml
skinparam defaultFontName sans-serif
skinparam shadowing false
skinparam sequenceMessageAlign center

participant "Application" as App
participant "CacheNotifier" as CN
participant "QueueingSegment\nListener" as QSL
participant "Cache\n(data container\n+ stores)" as DC

App -> CN: addListenerAsync(listener,\nincludeCurrentState=true)
activate CN

CN -> QSL: Create per-segment handler
activate QSL

CN -> DC: Iterate all entries\n(segment by segment)
activate DC

loop for each entry
  DC --> QSL: entry(key, value)
  QSL -> App: CacheEntryCreatedEvent\n(synthetic)
end

DC --> CN: iteration complete
deactivate DC

QSL -> QSL: Flush queued\nconcurrent updates
QSL -> App: Deliver queued events\n(in segment order)
deactivate QSL

CN --> App: registration complete
deactivate CN

note over QSL
  Concurrent cache updates that arrive
  during iteration are queued per segment.
  After the segment iteration completes,
  queued events are replayed to ensure
  no events are lost or duplicated.
end note
@enduml
....

The initial state transfer uses a `QueueingSegmentListener` that iterates over cache entries segment by segment.
Any concurrent cache updates that arrive during the iteration are queued.
Once a segment's iteration completes, the queued events for that segment are flushed to the listener.
This guarantees that the listener sees a consistent view: every entry is delivered exactly once, and no live events are lost during the transition.

For clustered listeners with initial state, all nodes participate: each node iterates its locally owned segments and sends the results to the origin node.
